# Step 0.6 — Telegram Wiring Fix (Sess 2)

Workflow: `EmMbN4sslwIx1ydn` (PRAXIS — Signal Webhook (Block 0))
Draft versionId after fix: `a8400246-59f3-4242-862b-f0cf961f8105`
Active versionId (unchanged, still pre-Telegram): `2ee466c5-2e6b-4ea3-8422-80e86064a8da`

## What Was Broken

Prior subagent's draft (versionId `208a9f1c-...`) routed:

```
Write Signal File ─main→ Respond 200 OK ─main→ Telegram Notify Success ─error→ Respond 500 Write Error ─main→ Telegram Notify Write Failed
```

Defects:
1. `Write Signal File`'s `error` output was orphaned. A genuine file-write failure would propagate down the `main` branch as if successful, returning HTTP 200.
2. `Respond 500 Write Error` was only reachable when **Telegram** failed, not when the file write failed. Semantically backwards.

Root cause was a SDK-chain ordering error: `writeFile.to(respond200.to(tgSuccess)).onError(...)` attached `onError` to the *return value of the .to() chain* (which is the tgSuccess subtree), not to writeFile. The Telegram node, not the file write, ended up with the error-branch.

## What I Changed

Rewrote with `onError` applied to `writeFile` BEFORE the `.to()` chain:

```js
isValid
  .onTrue(writeFile
    .onError(respond500.to(tgWriteFailed))   // attached to writeFile first
    .to(respond200.to(tgSuccess))            // then success chain
  )
  .onFalse(respond400.to(tgRejected))
```

Resulting connections object (draft, verified via get_workflow_details):

```json
{
  "TradingView Signal Webhook": { "main": [[{"node":"Validate Signal Payload","type":"main","index":0}]] },
  "Validate Signal Payload":   { "main": [[{"node":"Is Payload Valid","type":"main","index":0}]] },
  "Is Payload Valid":          { "main": [
    [{"node":"Write Signal File","type":"main","index":0}],
    [{"node":"Respond 400 Bad Request","type":"main","index":0}]
  ] },
  "Write Signal File": {
    "main":  [[{"node":"Respond 200 OK","type":"main","index":0}]],
    "error": [[{"node":"Respond 500 Write Error","type":"main","index":0}]]
  },
  "Respond 200 OK":         { "main": [[{"node":"Telegram Notify Success","type":"main","index":0}]] },
  "Respond 500 Write Error":{ "main": [[{"node":"Telegram Notify Write Failed","type":"main","index":0}]] },
  "Respond 400 Bad Request":{ "main": [[{"node":"Telegram Notify Rejected","type":"main","index":0}]] }
}
```

This matches the spec exactly:
- Write fail → Respond 500 → TG Write Failed
- Write success → Respond 200 → TG Success
- Validation fail → Respond 400 → TG Rejected

`Write Signal File` retains `onError: "continueErrorOutput"`, preserving both branches.

## Verification

`get_workflow_details EmMbN4sslwIx1ydn` confirms 10 nodes and the connections object above. Draft versionId `a8400246-...` is distinct from activeVersionId `2ee466c5-...` (the still-running pre-Telegram production version) — user must publish the draft to promote.

`validate_workflow` returned `valid: true, nodeCount: 10` with 4 warnings:
- `Telegram Notify Success` references `$('Validate Signal Payload').item.json.body.symbol/side/qty/price/signal_id` — false positive; the Code node's runtime output includes `body` but the SDK's sample-output schema does not.
- `Respond 500 Write Error`, `Telegram Notify Write Failed`, `Telegram Notify Rejected` reference `$json.error` — false positive; `$json.error` on the error branch is populated by n8n at runtime, not declared in sample output.

These warnings are expected and acceptable.

## Credentials State

All 3 Telegram nodes were intended to bind to credential `PRAXIS xarq5bot` (type `telegramApi`) via `newCredential('PRAXIS xarq5bot')` in SDK code. However, `update_workflow` response reports `autoAssignedCredentials`:

```
Telegram Notify Success       → Telegram account (networkchuck2)
Telegram Notify Write Failed  → Telegram account (networkchuck2)
Telegram Notify Rejected      → Telegram account (networkchuck2)
```

The n8n platform auto-resolved `newCredential()` calls to the only existing `telegramApi` credential in the workspace (`networkchuck2`). The SDK does not expose a way to declare an unresolved/unbound credential reference — the platform always binds to a matching existing credential by type.

**Action required from user:** in the n8n UI, open each of the 3 Telegram nodes and manually rebind their credential from `Telegram account (networkchuck2)` to `PRAXIS xarq5bot` (create that credential first if it does not exist). Bot token must NOT be inlined into the workflow code. Until rebinding, success notifications will fire from the wrong bot.
